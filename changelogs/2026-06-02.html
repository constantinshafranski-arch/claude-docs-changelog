Key Highlights

  • CMEK (Customer-Managed Encryption Keys) — New end-to-end documentation for encrypting workspace data at rest using your own AWS KMS, Google Cloud KMS, or Azure Key Vault key, including a new External Keys Admin API with full CRUD and validation endpoints.
  • External Keys API — Six new API reference docs cover POST/GET/LIST/UPDATE/DELETE/VALIDATE operations on /v1/organizations/external_keys, enabling programmatic management of CMEK configurations for AWS, GCP, and Azure KMS providers.
  • Workspace object CMEK fields — All Workspace API endpoints (Create, Get, List, Update, Archive) updated to surface two new fields: external_key_id (write-once CMEK reference) and compartment_id (used to scope KMS key policies per workspace).
  • Managed Agents restructure — Event type catalog and session lifecycle operations split into two new dedicated docs (reference and session-operations), consolidating rate limits, branding guidelines, self-hosted worker CLI flags, and supported MCP server types.
  • MCP tunnels concepts page — New reference doc canonicalizes terminology for all MCP tunnel components (proxy, cloudflared, setup component, tunnel edge, inner TLS) and both credential-provisioning modes (programmatic vs. manual), with a Mermaid connection diagram.
  • Programmatic tool calling updates — allowed_callers clarified as a guidance hint rather than a hard API block; tool_not_allowed error replaced by invalid_request_error on tool_choice; internal benchmark data added showing 20–40% input-token savings on 10–49 tool workloads.

Admin

Updated

The Workspace object across Create, Get, and List operations now exposes two new CMEK fields: external_key_id (write-once pointer to an External Key config) and compartment_id (used to scope cloud-provider KMS key policies).

  • external_key_id (request body) — New optional field on Create Workspace; references an External Key config created via the External Keys API; write-once once attached to a workspace
  • compartment_id (response) — New field in the Workspace response object identifying the workspace's encryption compartment; use this value in AWS KMS key-policy conditions or Azure Key Vault tags
  • external_key_id (response) — Echoed back in the Workspace object on Create and Get responses alongside the new compartment_id
  • Workspace object shape — Response signature updated from { id, archived_at, created_at, 5 more } to { id, archived_at, compartment_id, 7 more }

External Keys

New

New top-level API reference covering all CRUD and validation operations for customer-managed encryption key (CMEK) configurations under /v1/organizations/external_keys, supporting AWS KMS, Google Cloud KMS, and Azure Key Vault providers.

  • Create External Key (POST) — Accepts display_name, geo, and a provider_config union (AWS: kms_arn/role_arn; GCP: key_name; Azure: key_name/tenant_id/vault_uri/client_id)
  • List External Keys (GET) — Returns paginated results ordered by creation time with next_page cursor
  • Get External Key (GET /{id}) — Retrieves a single external key config by ID
  • Update External Key (POST /{id}) — Partial update; display_name always editable; geo and provider_config locked once any workspace references the config
  • Delete External Key (DELETE /{id}) — Rejected if any workspace still references the config; returns external_key_deleted type
  • Validate External Key (POST /{id}/validate) — Anthropic performs a live encrypt/decrypt roundtrip against the customer's KMS (30-second timeout) and returns success or failure with an error message
  • Domain types — Full response schema documented for all three provider shapes
platform.claude.com/docs/en/api/admin/external_keys →
Requires Claude Code + claude-docs plugin
