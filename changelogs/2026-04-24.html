Release Notes

Key Highlights

  • Memory Stores API (new) — Full CRUD API for `/v1/memory_stores` added across all SDKs (Python, TypeScript, Go, Java, Ruby, C#) and CLI, including sub-resources for individual memories, memory versions with list/retrieve/redact, and archive operations.
  • User Profiles beta removed — The `user-profiles-2026-03-24` beta flag and all associated `user_profiles` endpoints have been removed from the `AnthropicBeta` enum and API surface across all SDKs.
  • BetaCompactionBlockParam schema change — The `encrypted_content` field has been removed from `BetaCompactionBlockParam` (and `BetaCompactionBlock` in batch results), affecting Messages create, count_tokens, and batch endpoints.
  • Models API: xhigh thinking capability removed — The `xhigh` thinking-effort capability field was removed from `BetaModelInfo.thinking` capability support object in the Models list/retrieve endpoints.
  • Large-scale beta header updates — The `AnthropicBeta` union type was updated from 20 to 19 named members across 178+ API reference docs spanning all SDKs and CLI.
  • Many API reference entries omitted — Due to volume, individual SDK-specific pages (Python, TypeScript, Go, Java, Ruby, C#, CLI) for all updated and new endpoints were not enumerated separately; the changes described apply across all SDK variants.

Memory Stores

New

New top-level API resource at `/v1/memory_stores` providing full CRUD operations for memory stores, with nested sub-resources for individual memories (create, list, retrieve, update, delete) and memory versions (list, retrieve, redact), documented across all SDKs and the CLI.

  • POST /v1/memory_stores — New `CreateMemoryStore` endpoint accepting a name and metadata body, returning a `BetaMemoryStore` object.
  • GET /v1/memory_stores — New `ListMemoryStores` endpoint with `created_at[gte]`, `created_at[lte]`, `include_archived`, `limit`, and `page` query parameters.
  • GET /v1/memory_stores/{id} — New `GetMemoryStore` endpoint to retrieve a store by ID.
  • POST /v1/memory_stores/{id} — New `UpdateMemoryStore` endpoint for patching name and metadata.
  • DELETE /v1/memory_stores/{id} — New `DeleteMemoryStore` endpoint returning a deletion confirmation.
  • POST /v1/memory_stores/{id}/archive — New `ArchiveMemoryStore` endpoint to soft-delete a store.
  • Memories sub-resource — New `CreateMemory`, `ListMemories` (with depth/limit/order/page/path_prefix/filter parameters), `GetMemory`, `UpdateMemory`, and `DeleteMemory` (with `expected_content_sha256` guard) endpoints under each store.
  • Memory Versions sub-resource — New `ListMemoryVersions` (filterable by api_key_id, created_at range, memory_id, operation), `GetMemoryVersion`, and `RedactMemoryVersion` endpoints.
  • SDK coverage — All endpoints are documented for Python, TypeScript, Go, Java, Ruby, C# SDKs and the CLI beta namespace.

New

New nested resource under memory stores providing full lifecycle management of individual memory entries, including create, list, retrieve, update, and delete operations with versioning support.

  • POST /v1/memory_stores/{id}/memories — Creates a new memory entry; supports optional `view` query param (`basic` or `full`) controlling response verbosity.

  • GET /v1/memory_stores/{id}/memories/{memory_id} — Retrieves a single memory entry by ID with optional view parameter.
  • POST /v1/memory_stores/{id}/memories/{memory_id} — Updates a memory entry's content or metadata.
  • DELETE /v1/memory_stores/{id}/memories/{memory_id} — Deletes a memory entry; accepts optional `expected_content_sha256` for optimistic concurrency.
