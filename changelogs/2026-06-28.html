Advisor tool

Updated

The advisor tool's model compatibility matrix was expanded to include Claude Fable 5 and Mythos 5 as advisor options, the minimum advisor model was set to Claude Sonnet 4.6, equal-capability pairings were permitted, and result type behavior was clarified per model family.

  • Model compatibility table — Claude Fable 5 (claude-fable-5) and Claude Mythos 5 (claude-mythos-5) added as valid advisors for all executor tiers; executor rows for Fable 5 and Mythos 5 added
  • Minimum advisor requirement — advisor must now be Claude Sonnet 4.6 or a more capable model; models of equal capability (e.g., Opus 4.7 and Opus 4.8) can advise each other
  • Result type by model — Fable 5 and Mythos 5 advisors return advisor_redacted_result; all other compatibility-table models return advisor_result; stop_reason field presence now tied to whether max_tokens is set
  • Advisor inference details — clarified that the advisor runs under its own Anthropic-supplied system prompt and receives the executor's full transcript as quoted context, including text produced so far in the current turn
  • Multi-turn examples — TypeScript and C# code tabs added to the multi-turn conversation section alongside the existing Python example
  • Token output description — removed specific token-count estimates (400–700 text / 1,400–1,800 total) from the introductory description
  • TypeScript client initialization — const client = new Anthropic(); added to TypeScript code examples that were previously missing it

Programmatic tool calling

Updated

Container lifecycle semantics were tightened, the continuation request requirements were clarified (container ID now required, not optional), a full cURL example was added for the tool-result step, and async tool function signatures were described more precisely.

  • Container ID on continuation — container ID is now required (not optional) on any follow-up request that has pending programmatic tool calls; the API rejects the request without it
  • Container expiration timing — idle reclaim updated from 4.5 minutes to ~5 minutes; pending tool call timeout updated to ~4 minutes; expires_at timestamp described as the definitive signal
  • Step 3 cURL example — full cURL snippet added for the tool-result continuation request, showing message history, container field, and complete tools array
  • Tools array on continuation — explicitly documented that the same tools array from the original request must be resent; code execution tool must still be present
  • Async tool function signature — clarified that tools exposed to code as async Python functions take a single dict and return a string; Claude awaits them with top-level await and parses results with e.g. json.loads
  • allowed_callers version interoperability — both code_execution_20260120 and code_execution_20260521 are now accepted and interchangeable in allowed_callers; response blocks always tag the caller as code_execution_20260120
  • Java example — migrated to a void main() method entry point and updated to use Model.CLAUDE_OPUS_4_8 constant

Server tools

Updated

A major new section was added documenting how to handle turns where Claude calls both a server tool and a client tool in the same response, including detection, continuation format, and the distinction from pause_turn.

  • Mixed server + client tool turns — new section explains that when Claude calls a server tool and a client tool together, the API returns stop_reason: tool_use immediately; detect the deferred server call by finding a server_tool_use block with no matching result in the response
  • Continuation message format — follow-up user message must contain only tool_result blocks for the client tools; the same tools array must be kept, or the API returns a 400 naming the missing server tool
  • MCP tool equivalence — mcp_tool_use blocks from the MCP connector behave the same as server_tool_use blocks in this mixed-call pattern
  • Distinction from pause_turn — explicit comparison: pause_turn never leaves a client tool_use block waiting; mixed turns never have stop_reason: pause_turn
  • 400 error messages documented — exact API error strings for missing server tool result and missing client tool result added in a Warning block
  • Python, TypeScript, cURL, and CLI examples — full code examples added for the mixed web_fetch + run_command scenario across all major SDK languages
  • Page subtitle updated — subtitle now includes "mixed server and client tool turns" to reflect the new section

Tool Runner (SDK)

Updated

Tool runner documentation was updated with SDK-specific behavior clarifications for error logging, tool error interception hooks, C# state-mutation side effects, Ruby message-history takeover scoping, and Java strict-mode annotation.

  • Tool return types — return type guidance moved to the top of Basic usage; clarified that a returned string becomes a single text content block and structured data must be encoded as a string
  • Java strict mode — Java tool runner now marks tools with strict: true (strict tool use) derived from the class structure
  • Error logging by SDK — Python logs full exceptions (type + message + stack trace) via the logging module; Go, Ruby, C#, and PHP do not read ANTHROPIC_LOG and don't log failed tools automatically
  • Tool error interception hooks — generate_tool_call_response() (Python) and generateToolResponse() (TypeScript) are the only SDKs with this hook; other SDKs' tabs describe the closest alternative
  • C# state mutation warning — clarified that calling SetParams() or PushMessages() still runs the matched tools for that turn; a tool also run inside the loop body executes twice unless accounted for
  • Ruby message-history takeover scoping — takeover via runner.params[:messages] reassignment or feed_messages only occurs when called from inside an each_message or each_streaming block
  • Automatic compaction deprecation — Python, TypeScript, and Ruby client-side compaction noted as deprecated in favor of server-side context editing; Go, Java, C#, and PHP tool runners don't include client-side compaction
  • max_iterations — loop exit condition updated: runner also exits when it reaches max_iterations if set
  • Client initialization snippets — client = ... initialization lines added to all SDK tabs in the until_done and error-interception sections

Web fetch tool

Updated

Platform availability was clarified (Claude API, Claude Platform on AWS, and Microsoft Foundry; not Bedrock or Google Cloud), several error codes were renamed or added, and parameter documentation was tightened.

  • Platform availability — web fetch (with and without dynamic filtering) explicitly documented as available on Claude API, Claude Platform on AWS, and Microsoft Foundry; not available on Amazon Bedrock or Google Cloud
  • Dynamic filtering code execution — clarified that dynamic filtering uses the code execution tool, which the API enables automatically; users do not need to add it to the tools array
  • Error code updates — invalid_input renamed to invalid_tool_input; new url_not_in_prior_context error code added; url_not_allowed expanded to cover org-level domain settings and Anthropic-side restrictions; unsupported_content_type updated to include HTML explicitly
  • Error type field — error object type corrected from web_fetch_tool_error to web_fetch_tool_result_error in the example JSON
  • blocked_domains restriction — documented that blocked_domains cannot be combined with allowed_domains
  • max_uses and failed fetches — clarified that failed fetches count against the max_uses limit
  • max_content_tokens scope — noted that the limit applies to text content only, not binary content such as PDFs
  • Citations default — explicitly stated that citations are disabled by default for web fetch
  • PDF handling — clarified that PDFs are returned as base64-encoded data and processed like a directly attached PDF document
  • Cache bypass example — added a JSON snippet showing the use_cache: false parameter in context; added note that use_cache and response_inclusion are version-gated to web_fetch_20260309 and web_fetch_20260318 respectively
