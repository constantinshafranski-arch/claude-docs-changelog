Batch processing

Updated

Added an explicit constraint that each batched request must specify `max_tokens` of at least `1`, making cache pre-warming (`max_tokens: 0`) unsupported inside batches.

  • max_tokens minimum — Each request in a batch must now set max_tokens to at least 1; max_tokens: 0 is rejected.
  • Cache pre-warming incompatibility — Cache pre-warming (max_tokens: 0) is explicitly unsupported in batches because an ephemeral cache entry written during batch processing would likely expire before any follow-up request executes.

Prompt caching

Updated

Added a new "Pre-warming the cache" section documenting how to use `max_tokens: 0` to preload system prompts and tool definitions into the prompt cache before real user requests, with full multi-language code examples.

  • Cache pre-warming — New max_tokens: 0 API pattern runs the full prefill phase and writes cache entries without generating any output, returning an empty content array and stop_reason: "max_tokens".
  • Cache breakpoint placement — Documented that the cache_control breakpoint must be placed on the last shared block (e.g., system prompt or tool definitions), not on the placeholder user message, and that explicit cache breakpoints must be used instead of automatic caching.
  • Billing behavior — Pre-warm requests incur a standard cache write charge if the prefix is not already cached; zero output tokens are billed; usage.cache_creation_input_tokens confirms a write occurred.
  • Code examples — Added sample code for cURL, CLI, Python, TypeScript, C#, Go, Java, PHP, and Ruby demonstrating the pre-warm request pattern and expected response shape.
  • Typical usage pattern — Added guidance to fire a pre-warm request at application startup or on a scheduled interval, before real user requests arrive.
