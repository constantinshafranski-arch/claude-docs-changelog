Claude Code on Amazon Bedrock

Updated

AWS region resolution for Bedrock is now automatic as of v2.1.172, reading from the active AWS profile instead of requiring an explicit AWS_REGION, with GovCloud prefix support added for model IDs.

  • AWS_REGION — No longer required; as of v2.1.172 the region is resolved in order: AWS_REGION → AWS_DEFAULT_REGION → active AWS profile (credentials file then config file) → us-east-1 fallback, matching AWS SDK precedence
  • AWS profile — Active profile is determined by AWS_PROFILE if set, otherwise default; AWS_SHARED_CREDENTIALS_FILE and AWS_CONFIG_FILE can override default file paths
  • /status command — Now shows the resolved region and notes the source when it came from AWS config files or the default fallback
  • GovCloud model IDs — Documentation now notes that cross-region inference profile IDs in AWS GovCloud regions use the us-gov. prefix
  • Mantle endpoint — Region for Mantle is now resolved using the same multi-step precedence as Bedrock (v2.1.172+); earlier versions use AWS_REGION only

Model configuration

Updated

The availableModels allowlist now enforces restrictions across subagent models, the advisor model, and fallback chains, and the opusplan 1M context behavior and allowlist matching logic were updated.

  • availableModels scope — Allowlist now applies to subagent frontmatter model fields, the Agent tool's model parameter, the /agents model picker, CLAUDE_CODE_SUBAGENT_MODEL, and the advisorModel setting — not just the main session
  • Blocked model behavior — /model with a blocked model is rejected with an error; a blocked --model flag or ANTHROPIC_MODEL is silently ignored at startup; a blocked subagent or advisor override falls back to the inherited or default model
  • opusplan 1M context — On tiers where Opus is auto-upgraded to 1M context, opusplan now also receives the upgrade in plan mode; use opusplan[1m] to force 1M when not on an auto-upgrade tier
  • availableModels matching — Matching now checks the model alias, version prefix (e.g., claude-opus-4-8), or full model ID; the [1m] suffix is stripped before matching; provider-specific prefixes such as us.anthropic. are not stripped and must be listed explicitly

Analytics APIs

New

New guide that explains the difference between the Claude Code Analytics API (Admin API key, Claude Console organizations) and the Claude Enterprise Analytics API (Analytics API key, claude.ai organizations), with step-by-step provisioning instructions for each.

  • Claude Code Analytics API — Uses an Admin API key (sk-ant-admin01-...) created in Claude Console; covers daily per-user Claude Code metrics including sessions, lines of code, commits, pull requests, tool acceptance, and estimated cost by model
  • Claude Enterprise Analytics API — Uses an Analytics API key created in claude.ai by the primary owner; covers organization-wide engagement, adoption, user activity, project/skill/connector usage, and cost reports for Enterprise plans
  • Key interoperability — Admin API keys and Analytics API keys are not interchangeable; organizations using both platforms can provision both key types independently
  • Provisioning steps — Step-by-step instructions provided for both APIs, including role requirements, key creation locations, and example curl invocations
