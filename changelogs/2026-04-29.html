CLI reference

Updated

Clarified the behavior of the --init, --init-only, and --maintenance flags to reflect their new relationship with the Setup hook event and print mode.

  • --init flag — Redefined to run Setup hooks with the init matcher in print mode only (claude -p --init "query"); no longer starts interactive mode.
  • --init-only flag — Redefined to run Setup and SessionStart hooks with the startup matcher, then exit without starting a conversation.
  • --maintenance flag — Redefined to run Setup hooks with the maintenance matcher in print mode only (claude -p --maintenance "query"); no longer starts interactive mode.

LLM gateway configuration

Updated

Documented the attribution block Claude Code prepends to system prompts and introduced the CLAUDE_CODE_ATTRIBUTION_HEADER env var to suppress it for gateways with their own prompt caching.

  • Attribution block — Claude Code now prepends a short attribution block (client version + conversation fingerprint) to the system prompt; the Anthropic API strips it before processing so it does not affect first-party prompt caching.
  • CLAUDE_CODE_ATTRIBUTION_HEADER — Set to 0 to omit the attribution block when your gateway implements its own prompt cache keyed on the full request body.

Sandboxing

Updated

Added WSL1/WSL2 sandboxing limitations, clarified proxy TLS behavior, and strengthened the domain-fronting security warning.

  • WSL1 unsupported — Added note that WSL1 lacks the Linux namespace primitives required for sandboxing; users seeing Sandboxing requires WSL2 must upgrade or run without sandboxing.
  • WSL2 Windows binary restriction — Documented that sandboxed commands cannot invoke Windows binaries (e.g., cmd.exe, powershell.exe, paths under /mnt/c/) and recommends adding such commands to excludedCommands.
  • Proxy TLS inspection note — Added callout clarifying the built-in proxy enforces the allowlist by hostname without terminating or inspecting TLS traffic.
  • Domain fronting warning — Expanded security warning to explain that the proxy's hostname-only allow decision can be bypassed via domain fronting or similar techniques, and recommends a TLS-terminating custom proxy for stronger guarantees.
