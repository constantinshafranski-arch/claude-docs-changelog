Key Highlights

  • MessageDisplay hook — New hook event fires while assistant message text streams to screen, supporting displayContent output to replace rendered text without affecting transcript or Claude's context
  • Batch processing: server tools — All server tools (web search, web fetch, code execution, MCP connectors, advisor, tool search) now work in batch requests with a higher per-turn iteration limit and automatic pause_turn continuation support
  • Batch processing: unsupported parameters — Clarified that stream, speed, store, previous_thread_event_id, cache_hint, context_hint, max_tokens: 0, and research_preview_2026_02 are not valid in batch requests

Hooks reference

Updated

Added full documentation for the new MessageDisplay hook event, including input schema, output fields, displayContent behavior, and updated the lifecycle diagram, event table, matcher table, and decision-pattern table.

  • MessageDisplay event — New hook event documented; fires per batch of completed lines as an assistant message streams, or once per message in non-interactive/Agent SDK runs with final: true and the full message in delta
  • MessageDisplay input — Input schema includes turn_id, message_id, index, final, and delta fields for tracking progress through a streaming message
  • MessageDisplay output — Supports hookSpecificOutput.displayContent to replace rendered text on screen; transcript and Claude's context retain the original text
  • SessionStart: sessionTitle output — New sessionTitle field in hookSpecificOutput sets the session title (equivalent to /rename); applies only on startup and resume sources
  • SessionStart: reloadSkills output — New reloadSkills: true field triggers a re-scan of skill and command directories after SessionStart hooks complete, making hook-installed skills available in the same session
  • SessionStart input: session_title — New session_title input field carries the current session title so hooks can avoid overwriting an explicitly set title
  • Lifecycle diagram — Updated to include MessageDisplay as a display-only event that runs while assistant message text streams

Batch processing

Updated

Expanded what can be batched to include all server tools, documented unsupported parameters that return validation errors, and added a new section on server tools and the agentic loop in batch context.

  • Server tools in batches — All server tools (web search, web fetch, code execution, MCP connectors, advisor, tool search) are now explicitly listed as supported in batch requests
  • Unsupported parameters — New table documents parameters that cause validation errors in batch requests: stream: true, speed, store/previous_thread_event_id, cache_hint/context_hint, max_tokens: 0, and research_preview_2026_02: "active"
  • Server tools and the agentic loop — New section explains that the batch worker runs the same server-side agentic loop with more iterations per turn than synchronous requests; stop_reason: "pause_turn" results can be continued with a follow-up batch or synchronous request
  • Web search throttling in batches — Batch worker throttles web_search per organization automatically; retries are handled internally but may extend completion time for large web-search batches
  • FAQ updated — Corrected FAQ answer to list specific unsupported parameters rather than claiming full feature parity with Messages API
